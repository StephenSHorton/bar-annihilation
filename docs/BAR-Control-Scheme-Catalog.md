# Beyond All Reason — Complete Control Scheme Catalog

> **Purpose:** a complete inventory of BAR's unit-control / commanding scheme, to serve as the build checklist and porting spec for a Planetary Annihilation: TITANS client mod.
>
> **How this was built:** Primary pass = agents reading BAR's actual source (sparse clone of `beyond-all-reason/Beyond-All-Reason` @ `7403bb8` — `luaui/Widgets/*.lua` + `luaui/configs/hotkeys/*`). Secondary pass = an independent web/wiki cross-check. The two were reconciled; the cross-check's unique findings are in the Appendix. Source-grounded rows cite their `Source file`; engine orders cite the Recoil `CMD.*` / `GameCMD.*` id.
>
> _Generated 2026-06-25._

---

Beyond All Reason (BAR) runs on the **Recoil** engine (a Spring fork). Its unit-control scheme is split across two layers:

- **Engine commands (Recoil `CMD_*` / `GameCMD.*`).** These are the real orders the simulation understands: `CMD.MOVE`, `CMD.ATTACK`, `CMD.RECLAIM`, `CMD.FIRE_STATE`, `CMD.INSERT`, etc. They are issued through `Spring.GiveOrderToUnit(s)` / `SetActiveCommand`, persist in the unit's command queue, and are replicated/sync-safe. `GameCMD.*` IDs (e.g. `UNIT_SET_TARGET`, `WANT_CLOAK`, `FACTORY_GUARD`, `QUOTA_BUILD_TOGGLE`, `AREA_MEX`, `BLUEPRINT_PLACE`) are BAR-defined custom commands implemented by gadgets but issued exactly like engine commands.
- **Lua widgets (LuaUI, client-side).** Everything under `luaui/Widgets/*.lua`. Widgets read mouse/keyboard, draw overlays, and *translate gestures into engine commands*. Many "features" are pure UI (range rings, state icons, ghost previews) or behavioral automations (clean builder queue, hold-position defaults) that issue standard engine commands under the hood. A widget that owns no `CMD_*` shows "—" in the Engine cmd column.

**Where the binds come from.** Default keys live in real profile files under `luaui/configs/hotkeys/`. BAR ships five selectable presets (`keyboard_layouts.lua`): **Grid** (the modern default, `grid_keys.txt` — z/x/c/v spatial build grid, command verbs on letters), **Grid (60%)**, **Legacy** (`legacy_keys.txt` — historic TA/Spring scheme, direct `buildunit_*` keys), **Legacy (60%)**, and **Custom** (`uikeys.txt`). `default_keys.txt` is a shim that `keyload`s `legacy_keys.txt`. Profiles compose shared fragments (`chat_and_ui_keys.txt`, `num_keys.txt`, `gridmenu_keys.txt`) via the engine `keyload` directive, and `sc_*` scancode tokens are remapped per physical keyboard layout. Binds below are written **grid:** / **legacy:** where the two diverge; "Any+" matches the key regardless of modifier state; comma-separated keys are double/triple-tap chains.

> Notation: in binds, `sc_x` = the physical-position key x; `+Shift` on a command almost always means *queue*; a chain like `sc_b,sc_b` is a double-tap.

## Movement & Formations

The core mechanic is **CustomFormations2** (`cmd_customformations2.lua`): a right-mouse-button drag samples a freehand polyline into one positioned order per unit (line/arc formation), with optimal unit→slot assignment. It only ever acts on **RMB**; Spring's native Alt+LMB box/area formation and engine area-commands are left to the engine. (`move*` arrow/numpad actions are *camera* controls — see the Camera section.)

## Porting notes for a PA: TITANS client mod

Planetary Annihilation: TITANS exposes a client-side UI/scripting layer (Coherent/JS over a C++ sim) with an order/command API, but no Recoil-style engine. Grouping BAR's features by porting difficulty (PA side kept high-level — no invented PA API names):

**(a) Discrete order/state issuable through PA's command API — most portable.** These map to "select units, then issue command/state to the sim":
- Selection plumbing: control-group set/select/add/toggle/focus, clear-on-empty, commander cycle, idle-builder selection, autogroups, the `select`-style filtered selection (would be reimplemented as a JS filter over the unit list), Take/Share-units (if PA exposes give/share).
- Queue editing that's index/tag based: skip-current, cancel-last, front-insert/append (PA queues support insert/append + shift-queue), no-duplicate-orders, clean-builder-queue, preserve-commands-after-unload.
- Stances/states with direct equivalents or close ones: fire-state (PA has fire/hold-fire), patrol/attack-move, repeat, on/off where a unit supports it, hold-position via stance, factory rally/repeat, "default-state-on-creation" automations (hold-position/hold-fire defaults), stop-cancels-selfd, set-target-style priority (only if PA exposes a persistent target order — otherwise (d)).
- Behavioral automations that just watch events + issue standard orders: idle-builder guard-after-build, factory-assist-fix, guard-remove, nano assist-priority, only-fighters-patrol, auto-repeat/auto-cloak defaults.

**(b) Needs world-space drag/draw input (mouse-down → move → up sampling) — moderate; requires a custom input handler that emits many positioned orders.** PA can issue per-unit positioned orders, so the *output* is feasible; the work is capturing the gesture and computing the geometry:
- CustomFormations2 line/arc formations + Hungarian/NoX assignment, single-unit freehand path, formation queue/insert variants, formation rendering.
- Drag-build modes (line/area/grid/box/around), build spacing/facing-by-drag, blueprint stamping, build-split distribution, area-mex radius drag.
- Area commands by radius drag (reclaim/repair/capture/resurrect-style/area-attack/area-unload spread) and the Alt/Ctrl area-command target filters, smart-area-reclaim, area-reclaim-enemy.
- Minimap/PiP box-select and minimap drag-to-command.

**(c) Needs a custom UI overlay (Coherent/JS HUD) — moderate; pure presentation but substantial UI work.** PA's HTML/JS UI can host these, but each is a from-scratch overlay:
- Grid build menu (spatial keyboard grid + category keys + per-builder grid data + queue-quantity batching), legacy build menu, build bar, order/command card with state-light cycling, factory quota/preset panels.
- World/screen overlays: selection platters, group-number labels, idle "sleeping" icons, fire-state/repeat/self-D icons, build-placement buildability cells, ghost build-queue, range rings / AoE / scatter / reclaim-field highlights, transport load indicators, wait icons, formation dots.
- The keybind system itself: profiles, sc_ scancode + physical-layout remap, modifier prefixes, keychain double/triple-tap cycling, action dispatcher, chain/toggle actions — would be re-implemented in PA's own keybind/JS layer (PA has a JSON keymap, but not BAR's scancode/keychain semantics, so this is real work).

**(d) Depends on PA engine support / may not port cleanly.** Features that lean on Recoil/Spring engine internals PA may not expose:
- Engine-internal primitives whose semantics BAR relies on: `CMD.INSERT` by tag/index, `CMD.REMOVE` by stable tag, wait-family (`WAIT`/`GATHERWAIT`/`SQUADWAIT`/`DEATHWAIT`/`TIMEWAIT`), `RESTORE` terraform, persistent `SET_TARGET`/`CANCEL_TARGET`, `WANT_CLOAK`, trajectory/manual-fire/D-gun, stockpile, self-destruct timing — each only ports if PA has an analogous order; terraform/D-gun/cloak/stockpile in particular have no clean PA analog.
- Selection-volume manipulation (Ignore Self / TraceScreenRay override), live `GetSelectionBox`/`GetUnitsInScreenRectangle`, `TestBuildOrder` previews, resource-spot snapping (PA has no metal-spots model), faction blueprint substitution — all assume Spring engine queries.
- Camera/view/minimap layer (adjacent): camera state get/set, anchors, overview/megamap, minimap rotation, PiP render-to-texture, spectator lock-camera/Player-TV, joystick camera, camera shake, map-draw/marks — PA's camera and spectator systems differ enough that these would be re-conceived against PA's camera API rather than ported, and several (overview camera, minimap rotation, shockfront shake) may have no equivalent.
